# P2P sync setup guide

This guide covers enabling the P2P module in darktable, choosing a passphrase, trusting peers by key fingerprint, pairing the mobile app using the QR scanner, and understanding LAN/IP reachability.

---

## 1. Enable P2P sync in darktable

Open darktable's **Preferences** dialog (Edit → Preferences, or the gear icon) and navigate to the **P2P sync** tab.

Check **Enable P2P sync** at the top of the tab. This starts the `dt-p2p-daemon` background process and enables all sync features. Unchecking it stops the daemon immediately and suspends all P2P activity.

The rest of the settings panel is greyed out while P2P is disabled; it becomes active as soon as you check the box.

---

## 2. Set a passphrase

The passphrase is the root secret shared by every machine that should sync together. It derives the Ed25519 cryptographic identity and TLS certificate used for all peer connections, as well as the HTTP auth token checked on every request.

**All darktable instances that should sync with each other must use the same passphrase.**

### Entering the passphrase

In the **connection** section of the P2P sync tab, type your passphrase into the **passphrase** field. Click **show** to reveal what you have typed. The field is hidden by default.

Click **Apply & restart daemon** to save the passphrase and restart the daemon with the new identity. The **fingerprint** row will populate with the SHA-256 hash of your public key once the daemon has started.

### Choosing a good passphrase

- Use at least 16 characters and mix words, numbers, or symbols.
- A random word sequence (e.g. generated by a password manager) works well.
- Shorter or common passphrases reduce the security of the derived keys.
- The passphrase is stored in plain text at `~/.config/darktable/peer.pw` and `darktablerc`. Do not reuse a passphrase you use elsewhere.

### Where the passphrase is stored

| Location | Purpose |
|----------|---------|
| `~/.config/darktable/peer.pw` | Read by `dt_p2p_show_pairing_qr` to embed in the QR code |
| `darktablerc` (`plugins/p2p/passphrase`) | Read by the preferences dialog on next open |

The daemon itself receives the passphrase as the `--passphrase` command-line argument when darktable launches it.

---

## 3. Trust peers with different passphrases (peer.keys)

Peers that share the **same passphrase** are trusted automatically — no extra steps needed. Their derived public keys match because the passphrase is the same on all sides.

You only need to manage `peer.keys` if you want to sync with a peer that uses a **different passphrase** — for example, a colleague's darktable installation or a second machine you don't want to give the same secret.

### How peer trust works

When the daemon connects to a peer whose TLS certificate fingerprint is not in the derived trusted set and not in `peer.keys`, the connection is rejected. The daemon probes the peer's certificate separately and records it as a *candidate peer* in `~/.config/darktable/peer.candidates`.

### Accepting a candidate peer via the UI

1. Make sure both daemons are running and can reach each other on the network.
2. In the **reachable peers** section of the P2P sync tab, click **Refresh**.
3. Peers whose fingerprints have been seen but not yet trusted appear in the list.
4. Click **Accept** next to the peer you want to trust.

Accepting immediately adds the peer's fingerprint to the live trusted set and appends it to `~/.config/darktable/peer.keys`. The daemon starts syncing with the peer without a restart.

### Manually editing peer.keys

`~/.config/darktable/peer.keys` is a plain-text file with one SHA-256 fingerprint per line. You can edit it directly in the **trusted peer keys (peer.keys)** section of the P2P sync tab, or with any text editor.

```
# one fingerprint per line, comments start with #
a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2
```

Click **Apply & restart daemon** after editing the text area in preferences to save changes and reload the trusted set.

### Obtaining a peer's fingerprint

The daemon writes its own fingerprint to `~/.config/darktable/peer.fingerprint` at startup. The current value is shown in the **fingerprint** row of the preferences tab. Click **copy** to copy it to the clipboard.

#### Via the preferences UI

The **reachable peers** and **candidate peers** lists in the P2P sync tab each show a fingerprint alongside the peer's IP address and URL. This lets you visually match a physical device to its fingerprint before trusting it:

1. Make sure the peer's daemon is running and the two machines can reach each other on the network.
2. In the **reachable peers** section, click **Refresh**. Peers that have made contact appear with their `https://<ip>:17842` URL and fingerprint hash.
3. Candidate peers (seen but not yet trusted) appear in the **candidate peers** section, also with their IP and fingerprint.

You can compare the displayed fingerprint with the value shown on the peer machine (its own **fingerprint** row) to confirm the identity before clicking **Accept**.

#### Via OpenSSL (headless machines)

For servers or machines without the darktable GUI, extract the fingerprint directly from the live TLS certificate:

```bash
openssl s_client -connect <peer-ip>:17842 2>/dev/null \
  | openssl x509 -pubkey -noout \
  | openssl pkey -pubin -outform DER \
  | sha256sum \
  | awk '{print $1}'
```

This computes the SHA-256 hash of the SubjectPublicKeyInfo (SPKI) — the same value the daemon uses. Paste the output into `peer.keys` on the trusting machine.

> **Note:** This is the SPKI fingerprint, not the full-certificate fingerprint printed by `openssl x509 -fingerprint`. Make sure you use the pipeline above, not the `-fingerprint` flag.

## 4. Configure directories

In the **directories** section of the P2P sync tab:

| Setting | Purpose |
|---------|---------|
| **proxy directory** | Directory scanned for `.proxy.avif` files to announce and serve to peers. Set this to your darktable library root (e.g. `~/Pictures`). |
| **import directory** | Where proxy files downloaded from remote peers are placed when they don't exist locally. Only relevant on machines that receive proxies from other darktable instances. |

Click **Apply & restart daemon** after changing either directory.

---

## 5. Add static peers (peers.txt)

On the same LAN, darktable instances discover each other automatically via mDNS. No configuration is needed.

If mDNS is blocked (common on managed networks or across subnets), add the address of each peer machine to the **static peers (peers.txt)** section:

```
# one entry per line — bare IPs, IP:port, or full https:// URLs
192.168.1.108
192.168.1.50:17842
https://myserver.local:17842
```

Bare IP addresses default to port 17842. Click **Apply & restart daemon** to activate new entries.

The daemon also reads `~/.config/darktable/peers.txt` directly; edits to the text area in preferences write that file.

## 6. LAN/IP reachability and limitations

The daemon communicates over HTTPS on port **17842**. For the phone and desktop to sync, the phone must be able to open a TCP connection to the desktop on that port. This section describes when that works automatically and what to do when it does not.

---

### Same-subnet Wi-Fi (typical home network)

This is the ideal case and requires no manual configuration:

- The desktop daemon starts, obtains its LAN IP (e.g. `192.168.1.108`), and begins listening on port 17842.
- Both devices are on the same subnet, so the phone can reach the desktop directly.
- mDNS discovery announces the desktop's address to the phone automatically; no static peer entry is needed.
- After QR pairing the phone knows the desktop's address and connects immediately.

If discovery is slow, check that **Enable P2P sync** is checked and the **address** row in preferences is populated (not "(unavailable)"). That address is what the phone dials.

---

### AP isolation / managed/corporate Wi-Fi

Many enterprise access points and some consumer routers enable **client isolation** (also called AP isolation or wireless isolation). Isolated clients cannot communicate directly with each other even on the same SSID — only with the router.

Symptoms: mDNS doesn't discover the desktop; the phone's connection status stays amber or red even after pairing.

Fix: add the desktop's IP address as a static peer in **peers.txt** (section 5 above). This bypasses mDNS and lets the daemon dial the desktop directly. If client isolation is enforced at the router level rather than per-SSID, direct connections will also be blocked and you will need to route through a VPN (see below).

> **How to tell:** from the desktop, run `ping <phone-ip>` while the phone is on the same Wi-Fi. If the ping times out, isolation is active.

---

### Different subnets or VLANs

Multicast — used by mDNS — does not cross subnet or VLAN boundaries. Static peer entries work if the router has a route between the subnets and port 17842 is not blocked by firewall rules between them.

On the desktop, allow inbound TCP 17842 in your firewall:

```bash
# UFW
sudo ufw allow 17842/tcp

# firewalld
sudo firewall-cmd --add-port=17842/tcp --permanent && sudo firewall-cmd --reload

# iptables
sudo iptables -A INPUT -p tcp --dport 17842 -j ACCEPT
```

---

### Desktop firewall blocking the phone

Even on the same subnet, a host-based firewall on the desktop can silently drop the phone's connection attempts. The symptom is the same as AP isolation: the phone never connects despite being on the same network.

Check that no local firewall rule is blocking port 17842 (see commands above). If you are running a Linux firewall with a default-deny policy, add an explicit allow for the port before starting the daemon.

---

### NAT traversal — reaching the desktop from outside the LAN

At startup the daemon attempts to create a port mapping via UPnP (IGDv2, then IGDv1) or NAT-PMP so it is reachable from outside the local network. If successful, the external URL (e.g. `https://203.0.113.5:17842`) is included in the `/manifest` response and shown in the **address** row of preferences alongside the LAN URL.

The phone discovers the external URL automatically via the manifest and can use it when the LAN address is unreachable (e.g. while on mobile data).

UPnP/NAT-PMP must be enabled on your router and not blocked by your ISP. If neither succeeds and you need remote access, use a VPN (see below).

---

### VPN (Tailscale, WireGuard, OpenVPN)

A VPN assigns each device a stable virtual IP that is routable regardless of the physical network either device is on. This is the most reliable way to sync when the phone and desktop are on different networks (e.g. phone on mobile data, desktop at home).

**Tailscale** (recommended — zero config):

1. Install Tailscale on the desktop (`sudo apt install tailscale` or from tailscale.com).
2. Install Tailscale on the phone (Android/iOS app).
3. Sign both into the same Tailscale account.
4. In darktable, add the desktop's Tailscale IP (shown in `tailscale ip -4`) as a static peer in **peers.txt**:
   ```
   100.x.y.z
   ```
5. Re-pair the phone using the QR code so it learns the Tailscale address.

> Tailscale addresses are in the `100.64/10` CGNAT range. The daemon's LAN IP detection skips this range by default; the Tailscale interface is used for routing but the daemon advertises its LAN IP separately. Add the Tailscale IP explicitly as a static peer rather than relying on mDNS.

**WireGuard / OpenVPN:** works the same way — use the VPN interface IP as the static peer address.

---

### Phone tethering / mobile hotspot

If the desktop connects to the internet through the phone's hotspot, the phone's role reverses: the phone is the router and the desktop is the client. The phone's hotspot gateway IP is typically `192.168.43.1` (Android) or `172.20.10.1` (iOS). In this configuration:

- mDNS discovery usually works because both devices are on the hotspot subnet.
- If it does not, add the phone's hotspot IP as a static peer on the desktop.
- The desktop's darktable IP visible to the phone will be an address assigned by the hotspot DHCP (e.g. `192.168.43.x`). The daemon auto-detects this and advertises it correctly.

---

### Finding your desktop's current LAN IP

The **address** row in the P2P sync preferences tab always shows the IP the daemon is using. Alternatively:

```bash
# Linux — IP of the interface used to reach the default gateway
ip route get 1.1.1.1 | awk '/src/{print $7}'

# macOS
route -n get default | awk '/interface/{print $2}' | xargs ipconfig getifaddr
```

Give this address to anyone who needs to reach your daemon as a static peer.

## 7. Pair the mobile app using the QR scanner

The QR pairing flow transfers the passphrase, your machine's address, and any static peer addresses to the mobile app in a single scan. It is the recommended way to configure the phone.

### On the desktop (generate QR)

1. In the P2P sync tab, make sure **Enable P2P sync** is checked and the daemon is **running** (the daemon row shows "running").
2. The **fingerprint** and **address** rows must be populated — if they show "(unavailable — start daemon first)", click **Apply & restart daemon** first.
3. Click **pair QR** next to the fingerprint row.
4. A dialog opens showing a QR code. Keep it on screen.

The QR code encodes a `darktable://pair?d=<base64>` URL containing:

```json
{
  "v": 1,
  "pp": "<passphrase>",
  "fpr": "<fingerprint>",
  "peers": ["https://192.168.1.108:17842", …]
}
```

The passphrase and all known peer addresses are included. Your machine's own address (`peer.localurl`) is listed first.

> **Security note:** The QR code contains the passphrase in plain text. Do not display it where it could be photographed by untrusted parties.

### On the phone (scan QR)

1. Open **darktable mobile**.
2. Tap the **Pair** tab in the bottom navigation bar (middle icon).
3. The camera viewfinder opens with an orange aiming reticle. Grant camera permission if prompted.
4. Point the camera at the QR code on the desktop screen until it locks on.
5. A confirmation dialog appears showing the peer addresses extracted from the QR code.
6. Tap **Accept** to apply the passphrase and restart the phone's sync daemon.

The phone will connect to the desktop within a few seconds. The status banner at the top of the app turns green and shows the peer count when the connection is established.

### Alternative: deep link

If the phone and desktop are on the same network, you can also send the `darktable://pair?d=…` URL directly (e.g. by email or messaging app) and open it on the phone. The app handles the `darktable://` URL scheme and presents the same confirmation dialog without needing to scan a QR code.

### Entering the passphrase manually

If you prefer not to use the QR scanner, tap **Settings** on the phone and enter the passphrase directly in the **Shared Passphrase** field. Tap the field and type the passphrase, then tap **Restart** in the Daemon section to apply it. You may also need to add the desktop's address as a static peer in the **Static Peers** section if mDNS is not available.

## 8. Verify the connection

### Desktop

In the **peer connection status** section, click **Refresh**. Each known peer appears with a status indicator:

| Icon | Meaning |
|------|---------|
| ✓ (green) | Successfully synced — peer is reachable and trusted |
| ✗ (red) | Last sync attempt failed |
| ? (grey) | Peer is known but has not been attempted yet this session |

The **address** row shows this machine's HTTPS URL. Give this address to other machines that need to reach you as a static peer.

### Mobile

The status banner at the top of the app shows the connection state:

- **Red banner** — daemon is not running or the socket is not connected. Check Settings → Daemon status and tap Start or Restart.
- **Yellow/amber banner** — daemon socket connected but no peer syncs have succeeded yet. Wait a few seconds after restarting; the first sync takes 2–5 seconds.
- **Green banner** — at least one peer is reachable. The peer count is shown.

Tap **Settings → P2P Log** for a live log of all daemon activity, filterable by category (peer, xmp, proxy, preview, import).
